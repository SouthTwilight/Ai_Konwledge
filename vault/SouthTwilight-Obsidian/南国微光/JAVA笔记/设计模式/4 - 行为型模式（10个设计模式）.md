#### 一、责任链模式

> **定义：** 为了**避免请求发送者与多个请求处理者耦合在一起**，于是将所有请求的处理者通过前一对象记住其下一个对象的引用而连成一条链；当有请求发生时，可将请求沿着这条链传递，直到有对象处理它为止
> **意图：** 处理多权限层级对客户请求的处理，将多个处理类按一定顺序进行链接，使请求处理自行传递，在一定层级处进行处
> **优点：** 1、降低对象间的耦合度；2、增强系统可扩展性；3、增强给对象指派职责的灵活性；4、责任链简化对象间连接；5、责任分担，符合类的单一职责原则

###### 📌 通常应用场景： 
		1. 多个对象可以处理一个请求，但具体由哪个对象处理该请求在运行时自动确定。 
		2. 可动态指定一组对象处理请求，或添加新的处理者。 
		3. 需要在不明确指定请求处理者的情况下，向多个处理者中的一个提交请求。
###### 通常实现方式：
- 模式结构
    1. 创建抽象处理类，包含一个同类型私有变量及其 set 和 get 方法，用于指明下一个处理者对象；一个请求处理方法。
    ```java
    //抽象处理者角色
    abstract class Handler {
    	// 指明下一个请求处理者
        private Handler next;
        public void setNext(Handler next) { this.next = next; }
        public Handler getNext() { return next; }
        //处理请求的方法
        public abstract void handleRequest(Request request);
    }
    ```
    
    2. 创建具体处理类，不同级别处理类都要继承于抽象处理类，并依据各自处理层级不同进行连接形成责任链
    ```java
    class ConcreteHandlerJunior extends Handler {
        public void handleRequest(Request request) {......}
    }
    class ConcreteHandlerSenior extends Handler {
        public void handleRequest(Request request) {......}
    }
    ```
    3. 由服务端创建责任链，并将客户端传递请求信息传入处理责任链开端处
    ```java
    		//创建并组装责任链
            Handler handlerJunior = new ConcreteHandlerJunior().setNext(new ConcreteHandlerSenior());
            //传入客户端请求信息
            handlerJunior.handleRequest(clientRequset);
    ```

#### 二、命令模式

> **定义：** 将一个请求封装为一个对象，**使发出请求的责任和执行请求的责任分割开**。这样两者之间通过命令对象进行沟通，这样方便将命令对象进行储存、传递、调用、增加与管理
> **意图：** 分离 “方法请求者” 与 “方法实现者” 之间经常存在紧密的耦合关系，利于软件功能的扩展与维护
> **优点：** 1、通过引入中间件（抽象接口/命令对象）降低系统的耦合度；2、扩展性良好，增加或删除命令方便；3、可以实现宏命令——命令模式与组合模式结合，将多个命令装配成一个组合命令（宏命令）4、方便实现 Undo( 命令撤销 ) 和 Redo( 命令恢复 ) 操作；5、可以为现有系统增加额外功能

###### 通常实现方式：
- 模式结构
    1. 创建抽象命令接口类，需要包含执行命令的抽象方法，一般命名为 execute
    ```java
    // 抽象命令类
    interface Command {
        public abstract void execute();
    }
    ```
    2. 创建具体命令类，需要实现抽象命令接口类，并包括一个私有的命令实现类变量，并在 execute 方法中调用命令实现类的命令实现功能完成请求的命令
    ```java
    // 具体命令
    class ConcreteCommand implements Command {
    	// 私有的命令实现变量
        private Executor executor;
        ConcreteCommand(Executor executor) {
            this.executor= executor;
        }
        public void execute() {
    		......
            executor.action();
        }
    }
    ```
    3.  创建命令实现类，需要具备执行命令功能相关操作的方法，是具体命令业务的真正实现者
    ```java
    // 命令实现类
    class Executor {
    	// 命令执行方法
        public void action() {......}
    }
    ```
    4. 创建命令调用类，需要具备请求发起的作用，通常拥有很多命令对象，并通过访问命令对象执行相关请求。命令调用类不直接访问接受者。
    ```java
    // 命令调用者类
    class CommandInvoker {
    	// 含有一个私有的命令类，命令调用者一般为一对应多种命令，所以定为抽象命令类型
        private Command command;
        public CommandInvoker(Command command) {
            this.command = command;
        }
        public void setCommand(CommandInvoker command) {
            this.command = command;
        }
        public void call() {
    		......
            command.execute();
        }
    }
    ```
    5. 客户端根据需要创建含有指定命令的命令调用者 
    ```java
        Invoker commandCaller= new Invoker(new ConcreteCommand(anExecutor));
        commandCaller.call();
    ```

#### 四、中介者模式

> **定义：** 定义一个中介对象来封装一系列对象之间的交互，使原有对象之间的耦合松散，且可以独立地改变它们之间的交互。
> **意图：** 解决多个对象之间复杂的交互关系，将单对单导致集群上的**网状关系结构，** 通过一个单对多在集群上形成**星型关系结构**的中介者，将原有的复杂耦合对象解耦。
> **优点：** 1、符合迪米特法则，提高模块相对独立性和类的可复用性。2、降低类之间的耦合，提高系统的灵活性。

###### 通常实现模式：
- 模式结构
    1. 创建抽象中介者类，需要具备注册和转发业务对象信息等基本抽象方法
    
    ```java
    //抽象中介者
    abstract class Mediator {
    	// 注册业务对象信息
        public abstract void register(Business business);
    	// 转发业务对象请求
        public abstract void relay(Business business); 
    }
    ```
    
    2. 创建抽象业务类，需要一个受保护的抽象中介者对象，提供具体业务对象交互的抽象方法，实现所有相互影响的具体业务类的公共功能。
    
    ```java
    //抽象业务类
    abstract class Business {
        protected Mediator mediator;
        public void setMedium(Mediator mediator) {
            this.mediator = mediator;
        }
    	// 业务类抽象交互方法；接收/发送
        public abstract void receive();
        public abstract void send();
    }
    ```
    
    3.  创建具体中介者类，实现抽象中介者提供的所有方法
    
    ```java
    //具体中介者
    class ConcreteMediator extends Mediator {
        private List<Business> business= new ArrayList<Business>();
    	// 注册具体业务对象，将其保存至特定的区域/模块/数据库
        public void register(Business business) {......}
    	// 寻找业务对象注册保存中心，是否找到对应业务对象，找到后调用**业务对象提供的交互方法**进行业务交互
        public void relay(Business business) {......}
    }
    ```
    
    4. 创建具体业务类
    
    ```java
    //具体业务类1
    class ConcreteBusinessOne extends Business{
    	//接收处理函数，由中介者通过调用具体业务类对象调用该方法
        public void receive() {......}
    	//发送处理函数，由具体业务类对象调用继承的中介者对象进行转发请求
        public void send() {......}
    }
    //具体业务类2
    class ConcreteBusinessTwo extends Business{
    	//接收处理函数，由中介者通过调用具体业务类对象调用该方法
        public void receive() {......}
    	//发送处理函数，由具体业务类对象调用继承的中介者对象进行转发请求
        public void send() {......}
    }
    ```
    
    5.  客户端调用中介者进行转发操作
    ```java
    		Business businessOne = new ConcreteBusinessOne();
            Business businessTwo = new ConcreteBusinessTwo();
    		Mediator mediator = new ConcreteMediator()
            mediator.register(businessOne);
            mediator.register(businessTwo);
            businessOne.send();
            businessTwo.send();
    ```
    
###### 📌 Java 语言中针对各数据库驱动工作的 JDBC 设计中 ORM 框架部分，就是一个典型的中介者模式实现。

#### 五、备忘录模式

> **定义：** 在不破坏封装性的前提下，捕获一个对象的内部状态，并在该对象之外保存这个状态，以便以后当需要时能将该对象恢复到原先保存的状态。
> **意图：** 解决由于某种因素影响情况下造成用户需要将已变更的系统状态返回成未变更状态，即反悔。
> **优点：** 1、提供了一种可以恢复状态的机制；2、实现了内部状态的封装；3、简化发起者类

###### 通常实现模式：

- 模式结构
    1. 创建备忘录角色类，需要具备一个私有的系统变更状态配置对象（需要进行反悔回退操作的配置类），可以**获取/更改**状态配置类
    ```java
    //备忘录角色类
    class Memento {
    	// 私有的系统变更状态配置对象
        private ConfigFile configFile;
        public Memento(ConfigFile configFile) { this.configFile = configFile; }
    	// 变更系统状态配置类的方法
        public void setConfigFile(ConfigFile configFile) {......}
        public ConfigFile getConfigFile() {......}
    }
    ```
    
    2. 建发起者角色类，需要具备一个私有系统变更状态配置对象。需要可以进行创建备忘录和从备忘录恢复系统状态配置的功能。
    
    ```java
    //发起者角色类
    class Originator {
    	// 发起者私有配置对象
        private ConfigFile configFile;
        public void setConfigFile(ConfigFile configFile) {......}
        public ConfigFile getConfigFile() {......}
    	// 发起者创建备忘录和从备忘录回退状态
        public Memento createMemento() {
    		......
            return new Memento(configFile);
        }
        public void restoreMemento(Memento memento) {
    		......
            this.setConfigFile(memento.getConfigFile());
        }
    }
    ```

    3. 创建管理者角色类，需要具备保存和获取备忘录的功能。但不可对备忘录内容进行访问和修改。
    
    ```java
    //管理者角色类
    class Caretaker {
        private Memento memento;
        public void setMemento(Memento m) {......}
        public Memento getMemento() {......}
    }
    ```
    
    4. 客户端调用备忘录管理者和发起者进行系统备忘
    ```java
    		Originator originator = new Originator();
            Caretaker caretaker = new Caretaker();
    		// 只有发起者角色可以访问备忘录
            originator.setState(configFile_1);
            caretaker.setMemento(originator.createMemento()); //保存状态
            originator.setState(configFile_2);
            originator.restoreMemento(caretaker.getMemento()); //恢复状态
    ```

#### 六、观察者模式

> **定义：** 指多个对象间存在一对多的依赖关系，当一个对象的状态发生改变时，所有依赖于它的对象都得到通知并被自动更新。这种模式有时又称作发布-订阅模式；
> **意图：** 解决一个行为导致一个或多个用户需要作出改变的问题。主要实现产品有消息订阅器（MQ）等。
> **优点：** 1、降低了目标与观察者之间的耦合关系，两者之间是抽象耦合关系；2、目标与观察者之间建立了一套触发机制。

###### 通用实现方式：

- 模式结构
    1. 创建抽象业务类，需要具备一个观察者对象的聚集类，并具备将对象注册到聚集类和从聚集类删除的方法；提供一个由子类实现的通知具体
    ```java
    //抽象业务类
    abstract class BussinessTarget{
    	// 观察者对象的聚集类
        protected List<Observer> observers = new ArrayList<>();
        // 为聚集类增加观察者的方法
        public void add(Observer observer) {......}
        // 从聚集类删除观察者方法
        public void remove(Observer observer) {......}
    	// 具体子类实现通知特定观察者方法的业务类方法（业务类改变 --> 观察者）
        public abstract void notify(); 
    }
    ```
    
    2. 建具体业务类，实现抽象的通知方法，按照具体目标类的要求找需要的观察者类进行通知其他特定目标类。
    
    ```java
    // 具体目标类
    class SpecificBussinessTarget extends BussinessTarget{
    	// 根据具体业务类不同的需求和改变，进行通知特定依赖该业务的观察者
        public void notify() {......}
    }
    ```
    
    3. 创建抽象观察者类，需要提供根据目标业务类通知进行反映动作的方法
    
    ```java
    //抽象观察类
    interface Observer {
    	// 根据订阅的业务类通知后进行的反应方法
        void response(); 
    }
    ```
    
    4. 创建具体观察者类，实现抽象观察者类提供的通知响应方法即可
    
    ```java
    class ConcreteObserverOne implements Observer {
    	// 根据订阅的具体目标业务类的通知作出**相应的自身状态改变** / **执行特定的业务操作**
        public void response() {......}
    }
    // 一个具体业务类可能有多个观察者，每个观察者需实现自己的响应方法
    class ConcreteObserverTwo implements Observer {
    	// 根据订阅的具体目标业务类的通知作出**相应的自身状态改变** / **执行特定的业务操作**
        public void response() {......}
    }
    ```
    
    5. 客户端调用观察者模式；最主要步骤是：
	 **1. 创建需要的具体业务对象后；
	 2、添加需要的观察者对象给业务对象；
	 3、调用具体业务对象的通知方法**
    
    ```java
    		// 这几部可以根据具体场景封装起来，作为一个消息服务的内部核心实现或其他
    		BussinessTarget targetBussiness = new SpecificBussinessTarget();
            Observer obsOne = new ConcreteObserverOne();
            Observer obsTwo = new ConcreteObserverTwo();
            targetBussiness.add(obsOne);
            targetBussiness.add(obsTwo);
            targetBussiness.notify();
    ```

#### 七、状态模式

> **定义：** 对有状态的对象，把复杂的“判断逻辑”提取到不同的状态对象中，允许状态对象在其内部状态发生改变时改变其行为
> **意图：** 解决状态机和状态判断耦合性太高，封闭太大导致扩展困难的问题；将不同状态提取为一个状态对象，将状态改变导致行为变更的动作放置于状态对象内部自主实现可以有效解决上述问题
> **优点：** 1、结构清晰，不同状态行为分割开，满足“单一职责原则”；2、状态转换显示化，减少对象间的相互依赖；3、状态类职责明确，有利于程序的扩展。

###### 通用实现方式：

- 模式结构
    1. 获取环境参数属性（即状态机上下文等），所有状态变更都是对于当前环境状态的变更，每个系统都有自己的环境状态，需要更具自己系统的需求设计。
    ```java
    // 环境上下文参数属性类
    class Context {
    	// 普遍具有一个私有的环境状态对象
        private State state;
        // 定义并获取初始环境状态
        public Context() {......}
        // 设置新状态
        public void setState(State state) {......}
        // 获取当前状态
        public State getState() {......}
        **// 对状态变更请求进行处理
        public void Handle() {......}**
    }
    ```
    
    2. 创建抽象状态类，提供所有需要由具体状态子类实现的特定状态行为，如需状态流转，则一种状态应当至少具备一种行为。
    
    ```java
    // 抽象状态类
    abstract class State {
    	// 状态流转，第一个状态，需要向第二个状态进行转变
        public abstract void StateOneHandle(Context context);
    	// 第二种状态
    	public abstract void StateTwoHandle(Context context);
		// 第三种状态
    	public abstract void StateThereHandle(Context context);
    }
    ```
    
    3. 创建具体状态类，实现具体的状态变更
    
    ```java
    //具体状态类1号；可流转往2/3
    class ConcreteStateOne extends State {
    	// 提示状态流转错误
        public void StateOneHandle(Context context) {......}
    	// 流转状态至2，并提示流转成功到状态2
		public void StateTwoHandle(Context context){......}
    	// 流转状态至3，并提示流转成功到状态3
    	public void StateThereHandle(Context context){......}
    }
    //具体状态类2号；只可流转往3
    class ConcreteStateOne extends State {
		// 提示状态流转错误
        public void StateOneHandle(Context context) {......}
    	// 提示状态流转错误 
    	public void StateTwoHandle(Context context){......}
		// 流转状态至3，并提示流转成功到状态3 
		public void StateThereHandle(Context context){......}
    }
    //具体状态类3号；可流转往2，或结束状态流转
    class ConcreteStateOne extends State {
    	// 提示状态流转错误
	    public void StateOneHandle(Context context) {......}
		// 流转状态至2，并提示流转成功到状态2
    	public void StateTwoHandle(Context context){......}
    	// 提示状态流转错误
		public void StateThereHandle(Context context){......}
    }
    ```
    
    4. 客户端调用
    
    ```java
    		// 创建(获取)环境配置参数
    		Context context = new Context();
    		......    // 初始环境业务运行代码                      
    		// 将状态流转到状态一
            context.StateOneHandle();
    		......    // 状态一业务运行代码
    		// 将状态流转到状态二   
            context.StateTwoHandle();
    		......    // 状态二业务运行代码
    		// 将状态流转到状态三
            context.StateThereHandle();
    		......    // 状态三业务运行代码
    ```
    

#### 八、策略模式

> **定义：** 该模式定义了一系列算法，并将每个算法封装起来，使它们可以相互替换，且算法的变化不会影响使用算法的客户。
> **意图：** 解决实现某存在**多个算法/策略**的功能，跟行业环境和条件不同执行该功能的不同策略的问题
> **优点：** 1、使用策略模式可以避免使用多重条件语句；2、提供一系列可供重用的算法族，转移通用算法到父类中可避免重复代码；3、提供相同行为在不同条件下的不同实现；4、对开闭原则完美支持，可根据开发需求灵活增加新算法；5、实现了算法使用与算法实现分离

###### 通用实现方式

- 模式结构
    
    1. 创建抽象策略类，需要提供具体策略子类需要实现的方法接口
    
    ```java
    // 抽象策略类
    interface Strategy {
    	// 提供策略方法
        public void strategyMethod();    
    }
    ```
    
    2. 创建一系列具体策略类，需要实现抽象策略定义的接口，提供具体一系列算法实现
    
    ```java
    // 具体策略类1号
    class ConcreteStrategyOne implements Strategy {
    	// 策略实现的一种算法
        public void strategyMethod() {......}
    }
    // 具体策略类2号
    class ConcreteStrategyTwo implements Strategy {
    	// 策略实现的另一种算法
        public void strategyMethod() {......}
    }
    ```
    
    3. 创建策略引用( 环境 )类，持有一个策略引用，供予客户端使用
    
    ```java
    // 策略引用（环境）类
    class Context {
    	// 私有的策略类对象
        private Strategy strategy;
    	// 获取该策略类
        public Strategy getStrategy() {......}
    	// 变更策略对象的引用
        public void setStrategy(Strategy strategy) {......}
    	// 调用私有策略对象的策略实现
        public void strategyMethod() {
    		...... // 调用策略前准备
    		strategy.strategyMethod();
    	}
    }
    ```
    
    4. 客户端通过策略引用类使用不同策略
    
    ```java
    		Context context = new Context();
            Strategy strategyOne = new ConcreteStrategyOne();
            context.setStrategy(strategyOne);
            context.strategyMethod();
    
            strategyTwo = new ConcreteStrategyTwo();
            context.setStrategy(strategyTwo);
            context.strategyMethod();
    ```

#### 九、模板模式

> **定义：** 定义一个操作中的算法骨架，而将算法的一些步骤延迟到子类中，使得子类可以不改变该算法结构的情况下重定义该算法的某些特定步骤。
> **意图：** 在确定系统中某个服务实现的关键步骤，并且知道关键步骤的执行顺序时，缺少部分或全部步骤的具体实现时，使用模板模式提供一个规定好的关键步骤方法
> **优点：** 1、封装了不变部分，扩展可变部分；2、父类提取公共部分代码，便于代码复用；3、子类可以通过扩展方式增加相应功能，符合开闭原则

###### 通用实现方法

- 模式结构
    1. 创建抽象模板类，提供模板方法所需的基本方法，实现按照一定顺序调用基本方法的模板方法。
    #📌 **基本方法（包含）**： 
		    1. 抽象方法：在抽象类中声明，由具体子类实现 
		    2. 具体方法：在抽象类中已经实现，在具体子类中可以继承或重写它
		    3. 钩子方法：**判断的逻辑方法**和**可被子类重写的空方法**两种， 钩子方法：子类可以选择继承扩展或者选择不关注该方法，使用父抽象类默认**( 空 )**实现
    ```java
    //抽象模板类
    abstract class AbstractTemplate {
        //模板方法
        public void TemplateMethod() {
            specificMethod();
            abstractMethodOne();
            abstractMethodTwo();
    				hookMethodTwo();
        }
        // 抽象类中的具体方法
        public void specificMethod() {......}
        //抽象方法1
        public abstract void abstractMethodOne();
        //抽象方法2
        public abstract void abstractMethodTwo();
    	//钩子方法 
        public void hookMethodTwo(){};
    }
    ```
    
    2. 创建具体模板方法实现子类，实现抽象类中所定义的抽象方法和钩子方法，每个实现的具体方法都是模板方法这顶级逻辑中的一个组成步骤
    
    ```java
    //具体子类
    class ConcreteTemplate extends AbstractTemplate {
    	// 实现抽象方法
        public void abstractMethodOne() {......}
        public void abstractMethodTwo() {......}
    	// 重写并覆盖父类中空实现的钩子方法
    	public void hookMethodTwo(){......};
    }
    ```
    
    3. 客户端直接调用抽象类中的模板方法即可，无需关心具体实现
    
    ```java
    		// 使用具体模板创建抽象模板对象后调用模板方法即可
    		AbstractTemplate template = new ConcreteTemplate();
            template.TemplateMethod();
    ```

#### 十、访问者模式

> **定义：** 将作用于某种数据结构中的各元素的操作分离出来封装成独立的类，使其在不改变数据结构的前提下可以添加作用于这些元素的新的操作，为数据结构中的每个元素提供多种访问方式
> **意图：** 解决数据元素相对稳定但需要被多个访问者访问，且每个访问者按要求具备不同访问方式时，对数据进行不同处理方式的问题
> **优点：** 1、扩展性好，无需修改对象结构即可为其中元素添加新功能；2、复用性好，通过访问者定义数据元素的通用功能，提高系统复用程度；3、灵活性好，将数据结构与数据结构的操作进行解耦，操作集合可相对自由演化；4、单一职责原则，相关的访问行为被封装成一个访问者，每个访问者功能都比较单一

###### 通用实现方法

- 模式结构
    1. 创建抽象访问者接口，需要为每个不同的具体数据元素类提供一个访问接口。
    
    ```java
    //抽象访问者
    interface Visitor {
    	// 对具体数据元素1号提供访问访问接口
        void visit(ConcreteElementOne element);
    	// 对具体数据元素2号提供访问访问接口
        void visit(ConcreteElementTwo element);
    }
    ```
    
    2. 创建抽象数据元素接口，需要提供一个接收访问操作的接口方法，接受的访问者是该方法的参数。
    
    ```java
    //抽象元素类
    interface Element {
    	// 对访问者进行访问接受的方法
        void accept(Visitor visitor);
    }
    ```
    
    3. 创建具体访问者类，实现抽象访问者接口中对每类数据元素的访问方法，确定访问各类数据元素时的不同操作
    
    ```java
    // 微信端具体访问者
    class WeChatVisitor implements Visitor {
    	// 微信端访问具体数据元素1号时的操作，调用数据元素1号提供给微信的操作函数
        public void visit(ConcreteElementOne element) {
    		......
    		element.operationToWeChat();
    	}
    	// 微信端访问具体数据元素2号时的操作，调用数据元素2号提供给微信的操作函数
        public void visit(ConcreteElementTwo element) {
    		......
    		element.operationToWeChat();
    	}
    }
    // 浏览器具体访问者
    class WebVisitor implements Visitor {
    	// 浏览器访问具体数据元素1号时的操作，如上
        public void visit(ConcreteElementOne element) {......}
    	// 浏览器访问具体数据元素1号时的操作，如上
        public void visit(ConcreteElementTwo element) {......}
    }
    ```
    
    4. 创建具体数据元素类，实现抽象数据元素的接收访问操作方法，通常直接调用接受的访问者的访问方法，如 ***visitor.visit(this)*** 形式
    
    ```java
    //具体元素类1号
    class ConcreteElementOne implements Element {
        public void accept(Visitor visitor) {
            visitor.visit(this);   // 访问者模式核心代码语句
        }
    	// 提供浏览器访问1号元素的访问具体操作
        public Result operationToWeb() {......}
    	// 提供微信端访问1号元素的访问具体操作
    	public Result operationToWeChat() {......}
    }
    //具体元素类2号
    class ConcreteElementTwo implements Element {
        public void accept(Visitor visitor) {
            visitor.visit(this);   // 访问者模式核心代码语句
        }
    	// 提供浏览器访问2号元素的访问具体操作
        public Result operationToWeb() {......}
    	// 提供微信端访问2号元素的访问具体操作
    	public Result operationToWeChat() {......}
    }
    ```
    
   5. 创建对象结构类，是作为数据元素对象的聚集容器，并提供予以客户添加删除数据元素对象的操作方法，以及供予访问者对象遍历访问数据元素的方法
    
    ```java
    // 对象结构类
    class ObjectStructure {
    	// 数据元素聚集对象
        private List<Element> elementList = new ArrayList<Element>();
        // 供予访问者对象遍历访问数据元素的方法（可设计为按需求访问数据元素）
    		public void ergodicAccess(Visitor visitor) {
            Iterator<Element> element = elementList.iterator();
            while (element.hasNext()) {
                ((Element) element.next()).accept(visitor);
            }
        }
    	// 为数据元素聚集对象**添加或删除**单个数据元素对象
        public void add(Element element) {......}
        public void remove(Element element) {......}
    }
    ```
    
    6. 客户端调用对象结构类对象，根据需要添加或删除数据元素对象，并将访问对象传入给对象结构类对象的遍历访问方法进行每个**(或按需求访问)**数据元素的访问
    
    ```java
    		ObjectStructure objStructure = new ObjectStructure();
            objStructure.add(new ConcreteElementOne());
            objStructure.add(new ConcreteElementTwo());
            Visitor visitor = new ConcreteVisitorOne();
            objStructure.ergodicAccess(visitor);
            visitor = new ConcreteVisitorTwo();
            objStructure.ergodicAccess(visitor);
    ```
    

###### 📌 **注意：**
**具体数据元素类与具体访问者类**之间需相互配合，两者相互具有一定**耦合性**。且对各自的抽象类并不过于依赖，即依赖倒置被破坏
