#### 1、**适配器模式**

> **定义：** 将一个类的接口转换成客户希望的另外一个接口，使得原本由于接口不兼容而不能一起工作的那些类能一起工作 
> **意图：** 解决现有的组件库中已存在所需组件，但与当前系统接口规范不兼容，重新开发时成本很高等之类问题 
> **优点：** 客户端可透明调用目标接口；高复用；解耦目标类和适配者类

**常用实现方式**
1. 定义通用适配接口
2. 创建适配器类，继承通用适配接口
3. 适配器中引入目标类对象，并在通用适配接口方法中重写目标方法调用

#### **2、桥接模式**

> **定义：** 将抽象与实现分离，使它们可以独立变化；用组合关系代替继承关系实现，降低抽象和实现两个可变维度的耦合度。 
> **意图：解决某个类在两个或多个维度的变化产生的扩展困难的问题，即具有两种不同的分类属性（如：形状和颜色等） 
> 优点：** 抽象与实现分离，扩展能力强；符合开闭原则、合成复用原则；实现细节对外透明

**通常实现方式：**
<aside> ✏️ 将抽象化部分与实现化部分分开，取消二者的继承关系，改用组合关系

</aside>
- 创建具体实现角色
    1. 创建桥接接口
    2. 创建实现桥接接口的实现角色
    3. 客户端可创建具体角色为实现角色的桥接接口对象（面向接口编程）
    ```java
    //桥接接口（实现化角色）
    interface Implementor {
        public void OperationImpl();
    }
    //具体实现化角色
    class ConcreteImplementorA implements Implementor {
        public void OperationImpl() {
            System.out.println("具体实现化(Concrete Implementor)角色被访问");
        }
    }
    ```
- 扩展抽象角色（桥接接口扩展抽象类）
    1. 创建抽象化角色【即包含**特定类（接口）** 的抽象类】
    2. 抽象化角色中提供未实现的操作方法
    3. 扩展抽象化角色就是扩展抽象化角色提供的操作方法
    ```java
    //抽象化角色
    abstract class Abstraction {
        protected Implementor imple;
        protected Abstraction(Implementor imple) {
            this.imple = imple;
        }
        public abstract void Operation();
    }
    //扩展抽象化角色
    class RefinedAbstraction extends Abstraction {
        protected RefinedAbstraction(Implementor imple) {
            super(imple);
        }
        public void Operation() {
            System.out.println("扩展抽象化业务操作");
    				//特定类方法调用
            imple.OperationImpl();
        }
    }
    ```
     #📌 特定类一般为桥接接口，即实现角色对象；扩展抽象角色后可以得到综合实现角色和抽象角色的最终需要对象
- 客户端调用
    ```java
    // 创建具体实现角色
    Implementor imple = new ConcreteImplementorA();
    // 根据具体实现角色创建抽象扩展角色
    Abstraction abs = new RefinedAbstraction(imple);
    // 抽象扩展角色调用最终方法
    abs.Operation();
    ```
    

#### 3、组合模式

> **定义：** 是一种将对象组合成树状的层次结构的模式。_**用来表示“整体-部分”的关系，使用户对单个对象和组合对象具有一致的访问性。
> 意图：便于处理 ”整体-部分” 的关系，使树枝节点与叶子节点具有统一的访问特性
> 优点：** 客户端可以一致处理单个或组合对象；更容易在组合体内加入新对象，客户端无需关心新节点带来源代码的变更

<aside> 📌 即需要多重判断长流程的事件或过程，且流程可变时需要使用

</aside>

**通常实现方式：**
<aside> ✏️ 通常具有透明实现和安全实现两种方式： 1、透明实现：叶子结点与树枝节点都要实现组合业务接口，但不实现完整方法，以异常或空方法方式避免成为树枝节点。这样可以使客户端对树枝节点和叶子节点采取统一调用实现。 2、安全实现：只有树枝节点实现组合业务接口，叶子节点只负责叶子的职能即可。但需要客户端需要对叶子节点进行区分和特别处理。
</aside>

- **透明模式**
    1. 创建组合业务接口，需要提供进行节点管理方法和业务使用方法
    2. 树枝节点和叶子节点分别实现组合业务接口，但叶子节点不完成节点管理方法
    3. 客户端调用是统一创建为组合业务接口对象进行使用
    ```java
    //组合业务接口
    interface Component {
    		//节点管理方法
        public void add(Component c);
        public void remove(Component c);
        public Component getChild(int i);
    		//业务使用方法
        public void operation();
    }
    
    //树枝节点
    class Composite implements Component {
        private ArrayList<Component> children = new ArrayList<Component>();
        //实现节点管理方法
    		public void add(Component c) {
            children.add(c);
        }
        public void remove(Component c) {
            children.remove(c);
        }
        public Component getChild(int i) {
            return children.get(i);
        }
    		//实现业务使用方法
        public void operation() {
            for (Object obj : children) {
                ((Component) obj).operation();
            }
        }
    }
    
    //叶子节点
    class Leaf implements Component {
        private String name;
        public Leaf(String name) {
            this.name = name;
        }
    		//节点管理方法不完成
        public void add(Component c) {}
        public void remove(Component c) {}
        public Component getChild(int i) {return null;}
    		//只实现业务使用方法
        public void operation() {
            System.out.println("树叶" + name + "：被访问！");
        }
    }
    
    //客户端调用对其透明；只需要使用组合业务接口Component即可
    public static void main(String[] args) {
            Component c0 = new Composite();
            Component leaf1 = new Leaf("1");
            Component leaf2 = new Leaf("2");
            c0.add(leaf1);
            c0.add(leaf2);
            c0.operation();
    }
    ```
- **安全模式**
    1. 创建组合业务接口，只业务使用方法
    2. 树枝节点和叶子节点分别实现组合业务接口
    3. 树枝节点需要添加节点管理方法
    4. 客户端调用需要分别调用树枝节点和叶子结点进行处理
    ```java
    //组合业务接口
    interface Component {
    		//业务使用方法
        public void operation();
    }
    
    //树枝节点
    class Composite implements Component {
        private ArrayList<Component> children = new ArrayList<Component>();
        //添加节点管理方法
    		public void add(Component c) {
            children.add(c);
        }
        public void remove(Component c) {
            children.remove(c);
        }
        public Component getChild(int i) {
            return children.get(i);
        }
    		//实现业务使用方法
        public void operation() {
            for (Object obj : children) {
                ((Component) obj).operation();
            }
        }
    }
    
    //叶子节点
    class Leaf implements Component {
        private String name;
        public Leaf(String name) {
            this.name = name;
        }
    		//只实现业务使用方法
        public void operation() {
            System.out.println("树叶" + name + "：被访问！");
        }
    }
    
    //客户端调用对其透明；只需要使用组合业务接口Component即可
    public static void main(String[] args) {
            Composite c0 = new Composite();
            Component leaf1 = new Leaf("1");
            Component leaf2 = new Leaf("2");
            c0.add(leaf1);
            c0.add(leaf2);
            c0.operation();
    }
    ```
#### 4、装饰器模式

> **定义：** 在不改变现有对象结构的情况下，动态地给该对象增加一些职责（即增加其额外功能） 
> **意图：** 解决现存组件在不改变结构和核心功能时，动态扩展组件功能的问题
> **优点：** 继承的有力补充，比继承灵活，即插即用；使用不同装饰类或多种装饰类的排列组合，实现不同效果

**通常实现方式：**
- **模式实现概况**
    1. 抽象出组件核心功能，创建通用组件接口
    2. 目标组件实现通用组件接口，完成组件核心功能
    3. 创建抽象装饰类，包含一个私有通用组件接口类对象，通过构建方法进行赋值
    4. 创建具体装饰类，继承抽象装饰类
    5. 客户端使用时首先创建目标组件，然后根据目标组件创建被装饰后的组件
    ```java
    //抽象通用组件接口
    interface Component {
        public void operation();
    }
    
    //目标组件
    class ConcreteComponent implements Component {
    		//目标组件构建方法
        public ConcreteComponent() {......}
    		//组件核心功能
        public void operation() {......}
    }
    
    //抽象装饰类
    class Decorator implements Component {
        private Component component;
        public Decorator(Component component) {
            this.component = component;
        }
        public void operation() {
            component.operation();
        }
    }
    
    //具体装饰类
    class ConcreteDecorator extends Decorator {
        public ConcreteDecorator(Component component) {
            super(component);
        }
        public void operation() {
            super.operation();
            addedFunction();
        }
    		//装饰类为目标组件附加的装饰功能
        public void addedFunction() {......}
    }
    ```

#### 5、外观模式

> **定义：** 通过为多个复杂子系统提供一个一致接口，使众多子系统更容易被访问 
> **意图：** 解决复杂系统中客户端访问多子系统困难的问题，避免因系统内部子系统发生变化时客户端需跟随变化的问题 
> **优点：** 降低子系统与客户端的耦合度；减少客户端处理；降低大型系统编译依赖性

**通常实现方式：**
- 基础外观对象实现
    1. 创建各个子系统类
    2. 创建通用外观类，包含各子系统类的对象；并提供对外访问子系统的对外方法
    3. 客户端创建通用外观对象，通过访问外观对象提供的对外接口使用子系统方法
    ```java
    //子系统类
    class SubSystem_1 {......}
    //子系统类
    class SubSystem_2 {......}
    
    //外观对象
    class Facade {
        private SubSystem_1 subSystem_1 = new SubSystem_1();
        private SubSystem_2 subSystem_2 = new SubSystem_2();
        public void method() {
            subSystem_1.method();
            subSystem_2.method();
        }
    }
    
    //客户端调用所需两个子系统的方法
    Facade facade = new Facade();
    facade.method();
    ```
- **Java 中间件 (注解) 实现**
    1. 自定义所需注解作为切面
    ```java
    @Retention(RetentionPolicy.RUNTIME)
    @Target(ElementType.METHOD)
    public @interface DoDoor {......}
    ```
    2.  创建切面类，实现对客户端提供统一服务；切点定义为自定义注解 ；使用环绕切面编程
    ```java
    @Aspect
    @Component
    public class DoJoinPoint {
    	@Pointcut("@annotation(org.niexl.demo.design.door.annotation.DoDoor)")
        public void aopPoint() {}
    
    	@Around("aopPoint()")
        public Object doRouter(ProceedingJoinPoint jp) throws Throwable {......}
    	
    	......
    }
    ```
    
    3. 设置配置服务；将需要的提供的接口配置配置在文件中，便于动态的修改程序
    
    ```java
    @Configuration
    @ConditionalOnClass(StarterService.class)
    @EnableConfigurationProperties(StarterServiceProperties.class)
    public class StarterAutoConfigure {......}
    
    public class StarterService {......}
    
    @ConfigurationProperties("niexl.door")
    public class StarterServiceProperties {......}
    ```
    
    4. 客户端通过POM文件引入被封装的组件，通过中间件注解使用所需子系统服务
    
    ```java
    @DoDoor(key = "userId", returnJson = "{\"code\":\"1111\",\"info\":\"非白名单可访问用户拦截！\"}")
    @RequestMapping(path = "/api/queryUserInfo", method = RequestMethod.GET)
    public UserInfo queryUserInfo(@RequestParam String userId) {......}
    ```
    

#### 6、享元（Flyweight）模式

> **定义：** 运用共享技术来有效地支持**大量细粒度对象的复用**。它通过共享已经存在的对象来大幅度减少需要创建的对象数量、避免大量相似类的开销，从而提高系统资源的利用率。
>  **意图：** 解决面对对象程序设计中面临的要创建大量相同或相似对象实例的问题 
>  **优点：** 相同对象只要保存一份，这降低了系统中对象的数量，从而降低了系统中细粒度对象给内存带来的压力

 #要点 享元模式具有两个要求：细粒度和共享对象。细粒度导致对象数量多且性质相近，需要区分状  态为 **内部状态** 和 **外部状态**。  
1. **内部状态：** 对象共享出来的信息，存储在享元信息内部，并且不回随环境的改变而改变；如保存在连接池的连接对象中的用户名、密码、连接URL等信息  
2. **外部状态：** 是对象依赖的标记，随环境的改变而改变，不可共享；如每个连接被回收利用时，需要将它标记为可用状态，这是外部状态

**通用实现方式：**
- 模式结构
    1. 创建享元工厂，负责创建并管理享元角色；请求享元对象时，由工厂类检查并提供相应符合的享元对象，每个享元对象一个系统中应该只有一个
    2. 创建抽象享元角色，包含后续具体享元类需要实现的公共接口
    3. 创建具体享元类，继承抽象享元类
    4. **（非必须）**创建非享元对象，参数形式为享元角色传入非共享外部状态，如：可用/不可用
    
    ```java
    //抽象享元接口
    interface FlyweightFunction {
        public void operation(UnsharedFlyweight state);
    }
    
    //具体享元类
    class ConcreteFlyweight implements Flyweight {
        private String key;
    	// 享元构建器
        ConcreteFlyweight(String key) {......}
    	// 享元对象业务处理函数
        public void operation(UnsharedFlyweight outState) {......}
    }
    
    //享元工厂类
    class FlyweightFactory {
    	//创建享元管理列表
        private HashMap<String, Flyweight> flyweights = new HashMap<String, Flyweight>();
    	//管理享元对象，并返回符合要求的享元对象
        public Flyweight getFlyweight(String key) {
            Flyweight flyweight = (Flyweight) flyweights.get(key);
            if (flyweight == null) {
                flyweight = new ConcreteFlyweight(key);
                flyweights.put(key, flyweight);
            }
            return flyweight;
        }
    }
    
    //非享元角色
    class UnsharedFlyweight {
    	//外部参数信息
        private String info;
        UnsharedFlyweight(String info) {
            this.info = info;
        }
        public String getInfo() {
            return info;
        }
        public void setInfo(String info) {
            this.info = info;
        }
    }
    
	    //客户端调用享元模式处理的目标对象
    	FlyweightFactory factory = new FlyweightFactory();
    	Flyweight f01 = factory.getFlyweight("a");
    	Flyweight f02 = factory.getFlyweight("a");
    	//调用享元对象时传入非享元外部参数信息
    	f01.operation(new UnsharedConcreteFlyweight("第1次调用a。"));
    	f02.operation(new UnsharedConcreteFlyweight("第2次调用a。"))
    ```
    

### 7、代理模式

> **定义：** 由于某些原因访问对象不适合或者不能直接引用目标对象，代理对象作为访问对象和目标对象之间的中介对象给访问对象提供一个代理以控制对目标对象的访问。 
> **意图：** 解决客户不能或者不想直接访问目标对象使需要一个中介对象的问题 
> **优点：** 中介作用和保护目标对象；扩展目标对象功能；将客户端与目标对象分离，降低耦合度

**通用实现模式：**
- 模式结构
    
    1. 创建抽象主题类：声明所需目标业务方法
    2. 创建真实主题类：实现抽象主题中的业务方法，是代理对象最终引用的真实对象
    3. 创建代理类：提供与真实主题相同的接口，内部含对真实主题的引用，访问、控制或扩展真实主题功能
    4. 客户端创建代理对象，调用代理对象提供的增强/控制的业务方法
    
    ```java
    //抽象主题类
    interface Subject {
        void Operation();
    }
    
    //真实主题类
    class RealSubject implements Subject {
        public void Operation() {......}
    }
    
    //代理类
    class Proxy implements Subject {
    	//对真实主题的引用
        private RealSubject realSubject;
    	//对真实主题业务方法的访问控制或扩展
        public void Operation() {
            if (realSubject == null) {
                realSubject = new RealSubject();
            }
            preOperation();
            realSubject.Operation();
            postOperation();
        }
    	//对真实业务方法的预处理
        public void preOperation() {......}
    	//对真实业务方法的后续处理
        public void postOperation() {......}
    }
    
    //客户端调用相应的代理后的业务方法
    Proxy proxy = new Proxy();
    proxy.Request();
    
    ```
