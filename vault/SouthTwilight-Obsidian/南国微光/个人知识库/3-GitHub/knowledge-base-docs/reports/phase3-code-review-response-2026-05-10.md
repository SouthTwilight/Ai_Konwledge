## 二、逐条评议

### Bug 级别（3个）

#### B1. feishu_extractor.py 缺少 compute_hash() — **同意**

审查者定位准确，代码验证确认第 211-218 行确实没有调用 `compute_hash()`。其余 4 个提取器（RSS/Web/GitHub/Email）均有调用，属于遗漏。修复量 1 行。

#### B2. obsidian_writer.py 文件名冲突 counter 逻辑 — **同意，影响极小**

代码验证确认：`stem = filepath.stem` 在 while 循环内，导致第二次冲突时生成 `article-1-2.md` 而非 `article-2.md`。审查者的分析完全正确。

不过实际影响极低——同一天同一 slug 出现 3 次以上的概率极小。修复量 1 行（将 `stem = filepath.stem` 改为在循环外缓存 `original_stem`）。

#### B3. stats["errors"] 永远为 0 — **同意有问题，但"致命缺陷"用词过度**

代码验证确认：只有初始化 `stats["errors"] = 0` 和 `sys.exit(0 if stats["errors"] == 0 else 1)` 检查，中间无递增。

但审查者说这是"致命缺陷"，理由是"任务永远不会被标记为失败"。这个判断忽略了实际运行环境：

- **Hermes Cron 的失败判定**基于进程退出码和 `notify_on_complete`，而非 stdout 内容
- **管道自身有完善的文件日志**（`pipeline/logs/`），调试不依赖 stats 计数
- Cron 任务本身是 terminal() 调用，如果 Python 进程崩溃（未捕获的异常），进程退出码非零，Hermes 仍然会标记为失败

所以实际影响是：**stats 摘要报告不准确**（永远显示 0 errors），而非"自动化运维致命缺陷"。值得修，约 10 行。

### 风险级别（4个）

#### R1. 无 API 重试机制 — **部分同意**

审查者指出 L1/L2/L3 遇到 API 错误直接降级，建议增加 2-3 次指数退避重试。

**同意的部分**：2 次重试是合理的，能避免瞬时网络抖动导致文章处理失败。

**需要补充的设计上下文**：现有的降级策略不是"没有考虑失败"，而是**有意为之**——这是 Cron 定时任务，下一次运行（明天 8:00）会重新抓取同一篇文章。所以一次性失败的后果是**文章延迟一天处理**，而非永久丢失。加 2 次重试就够了，不需要复杂的重试框架。

#### R2. Email HTML→Markdown 正则脆弱 — **部分同意，审查者引用了不存在的文档**

**审查者声称**："README.md 也提到建议使用 markdownify"。

**事实**：查遍 README.md 全文，**不存在**任何关于 markdownify 的提及。审查者在这个点上不诚实，可能是"幻觉"生成的内容。

**实际情况**：当初选择正则而非 `markdownify`/`html2text` 是**有意的决策**——记录在项目 memory 中，原因是"Email extractor 不依赖 markdownify，用内置正则 HTML→MD 转换"。Newsletter 的 HTML 通常比较规范（不像网页那样复杂），正则覆盖了约 80% 的场景。

引入 `html2text` 作为 fallback 是合理的改进方向，但不是紧急问题。**不引入额外依赖**是项目的设计原则之一。

#### R3. trafilatura 重复调用 — **同意，纯优化**

代码验证确认 `web_extractor.py` 第 67-80 行确实对同一 HTML 调用了两次 `trafilatura.extract()`：一次获取纯文本，一次获取 JSON 元数据。

合并为一次 JSON 调用（从 JSON 中同时提取 content 和 metadata）是合理的优化。约 5 行改动。

#### R4. 配置路径解析丢弃目录结构 — **同意，使用场景有限**

代码验证确认 `config.py:138-140` 的 `path.name` 只保留文件名。如果用户传入 `--config ../my-configs/custom.yaml`，会被错误解析为 `pipeline/configs/custom.yaml`。

实际上项目运行至今所有 `--config` 参数都是 `pipeline/configs/` 下的文件名或绝对路径，这个 bug 从未触发。修起来 2-3 行。

### 代码质量（4个）

#### Q1. clusterer.py 死代码 GITHUB_API — **完全同意**

代码验证确认第 22 行 `GITHUB_API = "https://api.github.com"` 定义后从未使用。是从旧版 `github_extractor.py` 复制时遗留的残余。应删除。

#### Q2. clusterer.py 硬编码 Embedding API URL — **完全同意**

代码验证确认第 114 行 `https://open.bigmodel.cn/api/paas/v4/embeddings` 硬编码，而其他所有模块都通过 `ModelConfig.api_base` 读取。如果 API 地址变更或使用代理，clusterer 会单独失效。修起来约 5 行。

#### Q3. L1 API 失败默认评分 5 — **部分同意，fail-open 是刻意设计**

审查者建议改为 fail-closed 或可配置。

**需要补充的设计上下文**：评分 5 意味着文章进入 L2 处理（压缩摘要，不保留原文），token 消耗有限。如果改成 fail-closed（评分 0），则 API 故障期间**所有文章都被丢弃**——这违背项目核心原则"宁可重复不可遗漏"。

fail-open 的哲学是"宁可多处理几篇低质量文章，也不遗漏高价值文章"。但**同意应该可配置**，让用户在信任 API 稳定性后可以切换为 fail-closed。

#### Q4. email 白名单 endswith 逻辑宽松 — **同意，风险极低**

代码验证确认 `_is_sender_allowed()` 中 `sender_addr.endswith(allowed_lower)` 会导致 `"spoofexample.com"` 匹配 `"example.com"`。

理论风险确实存在，但实际场景中 DNS 域名注册不会出现这种碰撞。修起来简单（加 `"@" + allowed_lower`），值得做。

### M9 缺失评估 — **结论错误，需要反驳**

**审查者声称**：M9 是 Phase 3 计划中"唯一未实现的里程碑"，建议在下一轮迭代中列为首要任务。

**事实**：M9 经过了完整的四维评估框架分析（兼容性/修改量/维护成本/目标破坏），Playwright 被用户明确裁定**主动移除**，原因：

1. **Chromium 350MB 违背"轻量级单机工具"的设计哲学**——项目核心原则之一
2. **WSL 下 Cron 环境中 Playwright 进程泄漏风险高**——无头浏览器在无人值守环境中不稳定
3. **低频场景**——仅 SPA 页面需要 JS 渲染，投入产出比不合理
4. **标题回退链已作为 M9 的实际交付物实现**——`trafilatura → <title> → <h1> → URL slug`

M9 不是"未完成"，而是**范围裁剪后的完成状态**。用户裁决记录在案。README 路线图标 `[x]` 是正确的。

审查者在这里犯了两个错误：
- 没有读取 M9 评审记录和用户裁决上下文
- 没有验证 M9 的实际交付物（标题回退链 + URL 规范化增强）
